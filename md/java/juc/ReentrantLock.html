# Lock接口

Java SE 5之后并发包新增了Lock接口用来实现锁功能，它提供了与synchronized类似的同步功能，只是在使用时需要显示地获取和释放锁。

虽然它缺少了隐式释放锁的便捷性，但是却拥有了锁获取与释放的可操作性、可中断的获取锁以及超市获取锁等多种synchronized不具备的同步特性。

synchronized不具备的主要特性：

  • 尝试非阻塞地获取锁
  • 能被中断地获取锁
  • 超时获取锁

Lock是一个接口，定义了获取锁和释放锁的基本操作

public interface Lock {
    void lock();
    void lockInterruptibly() throws InterruptedException;
    boolean tryLock();
    boolean tryLock(long time, TimeUnit unit) throws InterruptedException;
    void unlock();
    Condition newCondition();
}

# 锁实现的基本原理

Sync的父类AbstractQueuedSynchronizer被称作队列同步器（AQS），这个类非常关键，它的父类是AbstractOwnableSynchronizer。

Lock锁将具备synchronized功能，可以阻塞一个线程，为了实现一把具有阻塞或环线功能的锁，需要几个核心要素：

  1. 需要一个state变量，标记该锁的状态。至少有两个值：0、1。对state变量的操作需要确保线程安全，也就是会用到CAS。
  2. 需要记录当前是哪个线程持有锁。
  3. 需要底层支持对一个线程进行阻塞或唤醒操作。
  4. 需要有一个队列维维护所有阻塞的先。这个队列也必须是线程安全的无锁队列，也需要用到CAS。

针对1、2分别在AbstractOwnableSynchronizer，AbstractQueuedSynchronizer类中体现：

public abstract class AbstractOwnableSynchronizer {
    /**
     * The current owner of exclusive mode synchronization.
     */
    private transient Thread exclusiveOwnerThread;
}

public abstract class AbstractQueuedSynchronizer extends AbstractOwnableSynchronizer {
    /**
     * The synchronization state.
     */
    private volatile int state;
}

tryAcquire是AQS的一个虚函数，再次尝试拿锁，被NonfairSync与FairSync分别实现。

    final boolean nonfairTryAcquire(int acquires) {
        final Thread current = Thread.currentThread();
        int c = getState();
        if (c == 0) { //无人持有锁，就开始抢锁
            if (compareAndSetState(0, acquires)) {
                setExclusiveOwnerThread(current);
                return true;
            }
        }
        else if (current == getExclusiveOwnerThread()) {
            int nextc = c + acquires;
            if (nextc < 0) // overflow
                throw new Error("Maximum lock count exceeded");
            setState(nextc);
            return true;
        }
        return false;
    }
    
    protected final boolean tryAcquire(int acquires) {
        final Thread current = Thread.currentThread();
        int c = getState();
        if (c == 0) {
            if (!hasQueuedPredecessors() &&
                compareAndSetState(0, acquires)) {
                setExclusiveOwnerThread(current);
                return true;
            }
        }
        else if (current == getExclusiveOwnerThread()) {
            int nextc = c + acquires;
            if (nextc < 0)
                throw new Error("Maximum lock count exceeded");
            setState(nextc);
            return true;
        }
        return false;
    }

# 阻塞队列与唤醒机制

没有抢到锁后就是用acquireQueued(addWaiter(Node.EXCLUSIVE), arg))将当前线程入队并阻塞线程。

首先addWaiter(Node.EXCLUSIVE)就是为当前线程生成一个Node,然后把当前线程放入队列尾部。

    //AQS
    private Node addWaiter(Node mode) {
        Node node = new Node(Thread.currentThread(), mode);
        // Try the fast path of enq; backup to full enq on failure
        Node pred = tail;
        if (pred != null) {
            node.prev = pred;
            //先尝试加到队列尾部，不成功执行enq
            if (compareAndSetTail(pred, node)) {
                pred.next = node;  
                return node;
            }
        }
        //enq内部会进行队列的初始化，新建一个空的Node。然后不断自旋直到将node加入队列尾部。
        enq(node);
        return node;
    }

阻塞线路的工作主要靠acquireQueued完成，线程一旦进入acquireQueued就会被无限期阻塞，即时有其他线程调用interrupt()函数也不能将其唤醒，除非当前线程拿到了锁。

    final boolean acquireQueued(final Node node, int arg) {
        boolean failed = true;
        try {
            boolean interrupted = false;
            for (;;) {
                final Node p = node.predecessor();
                if (p == head && tryAcquire(arg)) {
                    setHead(node);
                    p.next = null; // help GC
                    failed = false;
                    return interrupted;
                }
                if (shouldParkAfterFailedAcquire(p, node) &&
                    parkAndCheckInterrupt())
                    interrupted = true;
            }
        } finally {
            if (failed)
                cancelAcquire(node);
        }
    }

线程调用park，把自己阻塞起来，直到被其他线程环线，该函数返回。 park返回有两种情况：

  1. 其他线程调用了unpark
  2. 其他线程调用了t.interrupt

park与wait的作用类似，但是对中断状态的处理并不相同。如果当前线程不是中断的状态，park与wait的效果是一样的；如果一个线程是中断的状态，这时执行wait方法会报java.lang.IllegalMonitorStateException，而执行park时并不会报异常，而是直接返回。

所以，知道了这一点，就可以知道为什么要进行中断状态的复位了及为什么acquireQueued内部要是一个死循环了：

  • 如果当前线程是非中断状态，则在执行park时被阻塞
  • 如果当前线程是中断状态，则park会立即返回，然后parkAndCheckInterrupt方法会获取中断的状态，也就是true，并复位；
  • 再次执行循环的时候，由于在前一步已经把该线程的中断状态进行了复位，则再次调用park方法时会阻塞。

# unlock实现分析

unlock不区分公平还是非公平

    //Sync
    public final boolean release(int arg) {
        if (tryRelease(arg)) {
            Node h = head;
            if (h != null && h.waitStatus != 0)
                unparkSuccessor(h);
            return true;
        }
        return false;
    }

    protected final boolean tryRelease(int releases) {
        //每次调用锁的state-1
        int c = getState() - releases;
        //检查是否当前线程是锁的拥有者
        if (Thread.currentThread() != getExclusiveOwnerThread())
            throw new IllegalMonitorStateException();
        boolean free = false;
        //可以释放锁了
        if (c == 0) {
            free = true;
            setExclusiveOwnerThread(null);
        }
        //不需要使用CAS，因为是排他锁，只有一个线程可以调用unlock
        setState(c);
        return free;
    }

# lockInterruptibly实现分析

lockInterruptibly基本和lock过程一致，区别是在收到中断信号后不再阻塞，直接抛异常

    private void doAcquireInterruptibly(int arg)
        throws InterruptedException {
        final Node node = addWaiter(Node.EXCLUSIVE);
        boolean failed = true;
        try {
            for (;;) {
                final Node p = node.predecessor();
                if (p == head && tryAcquire(arg)) {
                    setHead(node);
                    p.next = null; // help GC
                    failed = false;
                    return;
                }
                if (shouldParkAfterFailedAcquire(p, node) &&
                    parkAndCheckInterrupt())
                    //区别在这里
                    throw new InterruptedException();
            }
        } finally {
            if (failed)
                cancelAcquire(node);
        }
    }

# Condition使用场景

以ArrayBlockingQueue为例。这是一个数组实现的阻塞队列，执行put的时候如果队列满了生产者线程被阻塞。执行take操作的时候，队列为空，消费者线程被阻塞。

public class ArrayBlockingQueue<E> extends AbstractQueue<E>
        implements BlockingQueue<E>{

    //核心是一把锁+两个条件
    final ReentrantLock lock;

    private final Condition notEmpty;

    private final Condition notFull;

    public ArrayBlockingQueue(int capacity, boolean fair) {
        if (capacity <= 0)
            throw new IllegalArgumentException();
        this.items = new Object[capacity];
        //构造的时候new 一把锁+两个条件
        lock = new ReentrantLock(fair);
        notEmpty = lock.newCondition();
        notFull =  lock.newCondition();
    }

    public void put(E e) throws InterruptedException {
        checkNotNull(e);
        final ReentrantLock lock = this.lock;
        lock.lockInterruptibly();
        try {
            while (count == items.length)
                notFull.await(); //队列满了，阻塞与notFull条件
            enqueue(e);
        } finally {
            lock.unlock();
        }
    }

    private void enqueue(E x) {
        final Object[] items = this.items;
        items[putIndex] = x;
        if (++putIndex == items.length)
            putIndex = 0;
        count++;
        //入队成功，队列有数据了，通知notEmpty条件
        notEmpty.signal(); 
    }

    public E take() throws InterruptedException {
        final ReentrantLock lock = this.lock;
        lock.lockInterruptibly();
        try {
            while (count == 0)
                //队列空了，阻塞于notEmpty条件
                notEmpty.await();
            return dequeue();
        } finally {
            lock.unlock();
        }
    }

    private E dequeue() {
        final Object[] items = this.items;
        @SuppressWarnings("unchecked")
        E x = (E) items[takeIndex];
        items[takeIndex] = null;
        if (++takeIndex == items.length)
            takeIndex = 0;
        count--;
        if (itrs != null)
            itrs.elementDequeued();
        //出队成功，通知notFull条件
        notFull.signal();
        return x;
    }
}

# await 实现分析

    public final void await() throws InterruptedException {
        //执行阻塞操作前先判断是否收到过中断信号
        if (Thread.interrupted())
            throw new InterruptedException();
        //加入Condition的等待队列
        Node node = addConditionWaiter();
        //关键，阻塞前需要先释放Lock的锁，否则会死锁，同Synchronized一样
        int savedState = fullyRelease(node);
        int interruptMode = 0;
        //自己不在同步队列中时进入循环，第一次会直接进入
        while (!isOnSyncQueue(node)) {
            //阻塞自己
            LockSupport.park(this);
            //被唤醒或者中断，才能继续往下走
            if ((interruptMode = checkInterruptWhileWaiting(node)) != 0)
                break;
        }
        //重新拿锁
        if (acquireQueued(node, savedState) && interruptMode != THROW_IE)
            interruptMode = REINTERRUPT;
        if (node.nextWaiter != null) // clean up if cancelled
            unlinkCancelledWaiters();
        //被中断唤醒，向外抛出异常
        if (interruptMode != 0)
            reportInterruptAfterWait(interruptMode);
    }

# notify实现分析

    public final void signal() {
        //只有持有锁的线程才能调用signal
        if (!isHeldExclusively())
            throw new IllegalMonitorStateException();
        Node first = firstWaiter;
        if (first != null)
            //唤醒队列中第一个线程
            doSignal(first);
    }

    private void doSignal(Node first) {
        do {
            if ( (firstWaiter = first.nextWaiter) == null)
                lastWaiter = null;
            first.nextWaiter = null;
        } while (!transferForSignal(first) &&
                    (first = firstWaiter) != null);
    }

    final boolean transferForSignal(Node node) {
        /*
         * If cannot change waitStatus, the node has been cancelled.
         */
        if (!compareAndSetWaitStatus(node, Node.CONDITION, 0))
            return false;

        //关键，CAS成功后，先把Node放入AQS的同步队列中，再执行unpark操作
        Node p = enq(node);
        int ws = p.waitStatus;
        if (ws > 0 || !compareAndSetWaitStatus(p, ws, Node.SIGNAL))
            LockSupport.unpark(node.thread);
        return true;
    }
