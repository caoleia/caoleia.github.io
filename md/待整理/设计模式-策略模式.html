对象行为型 定义

策略模式定义了一系列的算法，并将每一个算法封装起来，而且使它们还可以相互替换，策略模式让算法独立于使用它的客户而独变化。

网上对于各种模式的定义着实太过抽象，没有学习过的话根本不理解是什么意思。有很多模式经过多次学习还是没有什么印象，还是没有真正的理解其思想的精髓与使用的场景。

本次设计模式系列就是希望进行一次认真的总结，我会用自己的理解方式来表述对设计模式的学习和认识。

或许这样描述策略模式更加恰当

其用意是针对一组算法，将每一个算法封装到具有共同接口的一个独立的类中，客户端使用这个独立的类来执行算法。策略模式使可以将客户端选择算法的逻辑转移到独立的类中，可以在不影响到客户端的情况下发生变化。

# UML

  1. —Strategy 抽象策略角色： 策略类，通常由一个接口或者抽象类实现。
  2. —ConcreteStrategt 具体策略角色：包装了相关的算法和行为。
  3. —Context 环境角色：持有一个策略类的引用，最终给客户端调用。

# 转移条件语句

31天重构中提到策略模式最适合的场景：switch或if语句块很大，并且会随时引入新的判断条件。这时，最好使用策略模式将每个条件封装到单独的类中

基本的策略模式只是把各if中需要执行的算法进行了封装，如果将每个条件封装到Context中就避免了if判断的重复出现,如果有新的条件增加，只需要在Context中修改就可以了。

环境角色：

public class Context{
    private StrategyInterface strategy;

    public Context(... condition){
        switch (condition)
        {
            case: ...
                this.strategy =  new ConcreteStrategt1();
            case: ...
                this.strategy =  new ConcreteStrategt2();
            ...     
        } 
    }
    public void excute(){
        strategy.excute();
    }
}

# 消除条件语句

各种代码重构书籍中都将大段的switch、if块视为代码的坏味道，对于一些简单条件，可以使用字典策略来消除条件语句。 条件枚举：

public enum State
{
    contion1,
    contion2
}

环境角色：

public class Context{
    private static Map<State, StrategyInterface> StateMap = new HashMap()
    {
         { State.contion1, new ConcreteStrategt1() },
         { State.contion2, new ConcreteStrategt2() }
    };

    private StrategyInterface strategy;

    public Context(State state){
        this.strategy = StateMap[state];
    }
    public void excute(){
        strategy.excute();
    }
}

# 总结好处

  1. 策略模式提供了管理相关的算法族的办法，策略类的等级结构定义了一个算法或行为族，恰当使用继承可以把公共的代码移到父类里面，从而避免重复的代码。
  2. 策略模式使得算法的使用者就和算法本身分离开来，使得系统易于扩展。
